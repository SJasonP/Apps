# Internationalization

## Supported Locales

The site must fully support:

```text
en-US
zh-Hans
```

`en-US` is the fallback locale.

## Locale Selection

The frontend selects the locale from the browser language list.

Selection rules:

- If any browser language is `zh` or starts with `zh-`, use `zh-Hans`.
- Otherwise use `en-US`.

The selected locale sets:

- Visible UI text
- App product content
- Support and privacy page content
- Status labels
- Accessibility labels
- `document.documentElement.lang`
- `document.title`

## URL Policy

Locale is not represented in the URL.

Do not add locale prefixes such as:

```text
/en-US/history-lib
/zh-Hans/history-lib
```

Canonical page URLs remain language-neutral:

```text
/history-lib
/folders-guard
```

## Content Rules

Every user-facing string must have both `en-US` and `zh-Hans` content before it is used in the UI.

This includes:

- Navigation labels
- Buttons
- Section headings
- App descriptions
- Feature descriptions
- Support FAQ
- Privacy policy copy
- Status labels
- Accessibility labels

Brand names, app names, platform names, URLs, file extensions, and command names may stay unchanged when translation would reduce clarity.

Locked terms:

- `SJasonP` must stay `SJasonP` in every locale.
- `App` and `Apps` must stay `App` and `Apps` in every locale.

## Implementation

The current implementation keeps localization data in:

```text
src/src/i18n/enUS.ts
src/src/i18n/zhHans.ts
src/src/i18n/types.ts
src/src/i18n/index.ts
src/src/content/apps/{appName}/enUS.ts
src/src/content/apps/{appName}/zhHans.ts
```

The app uses a small typed dictionary instead of adding a runtime i18n framework dependency.
